# Responses API – ModelResponse Library

## Overview

This library provides a **schema-native, type-safe, and production-ready Java implementation** of the OpenAI **Responses API** `ModelResponse` payload. It enables consuming responses from the API for **single-turn, multi-turn, and multimodal conversations**, while preserving the exact object hierarchy defined in the schema.

Key design principles:

1. **Schema fidelity** – All classes match the official API definitions.
2. **Type safety** – Each output type, message, and content item is fully typed.
3. **Polymorphic outputs** – Supports multiple output types such as text, reasoning, code, and tool calls.
4. **Multimodal support** – Handles text, images, files, web searches, and tool outputs.
5. **Client-side context management** – Responses can reference `previous_response_id` or provide the full conversation timeline for determinism.

---

## Conceptual Model

### **ModelResponse**

* Top-level response object returned from the Responses API.
* Key fields:

| Field    | Type                | Description                                                      |
| -------- | ------------------- | ---------------------------------------------------------------- |
| `output` | `List<ModelOutput>` | Ordered list of outputs generated by the model for this request. |

> Each entry in `output` represents **one logical output unit** produced by the model, e.g., a text message, a tool call, or a reasoning object.

---

### **ModelOutput**

Represents **one item of the model’s output**. It is **polymorphic** via `type`:

| Type                                            | Description                                          |
| ----------------------------------------------- | ---------------------------------------------------- |
| `message` (`OutputMessage`)                     | Textual or multimodal content produced by the model. |
| `function_call` (`FunctionToolCall`)            | Indicates the model wants to invoke a function.      |
| `code_interpreter_call` (`CodeInterpreterCall`) | Requests code execution.                             |
| `web_search_call` (`WebSearchCall`)             | Requests a web search action.                        |
| `file_search_call` (`FileSearchCall`)           | Requests a file search action.                       |
| `mcp_call` (`McpCall`)                          | Specialized tool call (knowledge connector).         |
| `custom_tool_call` (`CustomToolCall`)           | Custom client-defined tool invocation.               |
| `reasoning` (`Reasoning`)                       | Model-provided reasoning, including sub-steps.       |

> `type` is the discriminator field used for deserialization. Each subclass has fields specific to the output type.

---

### **OutputMessage**

Represents **textual or multimodal content** returned by the model.

* Fields:

| Field     | Type                         | Description                                               |
| --------- | ---------------------------- | --------------------------------------------------------- |
| `content` | `List<OutputMessageContent>` | Array of content items (`text` or `refusal`).             |
| `role`    | `Role`                       | Typically `assistant` (or `system`, `developer`, `tool`). |

---

### **OutputMessageContent**

* Polymorphic content items inside a message:

| Type                                       | Description                                                                   |
| ------------------------------------------ | ----------------------------------------------------------------------------- |
| `output_text` (`OutputMessageContentText`) | Plain text output. May include `text`, optional `logprobs`, or `annotations`. |
| `refusal` (`OutputMessageContentRefusal`)  | Indicates the model refused to produce output.                                |

> The model may produce multiple `OutputMessageContent` items per message, e.g., text plus reasoning annotations.

---

### **Polymorphic Tool Outputs**

* Tool calls are **fully typed**, each representing an actionable request or client-side tool:

| Class                 | Purpose                             |
| --------------------- | ----------------------------------- |
| `FunctionToolCall`    | Model wants a function executed.    |
| `CodeInterpreterCall` | Model requests code execution.      |
| `WebSearchCall`       | Model requests a web search action. |
| `FileSearchCall`      | Model requests a file lookup.       |
| `McpCall`             | Knowledge connector query.          |
| `CustomToolCall`      | Any user-defined custom tool.       |

* Each tool call may have a **payload object**, e.g., `FunctionToolCall.arguments`, `WebSearchActionSearch.query`, etc.
* Tool outputs are fed back into the **next `ModelInputItem`** if needed for multi-turn workflows.

---

### **Logical Data Flow**

1. Client sends a `ModelRequest` to the API.
2. The model produces a **list of outputs** (`List<ModelOutput>`) in `ModelResponse.output`.
3. Each output may be:

   * A **message** with `OutputMessageContent` items (text, refusal).
   * A **tool call** requiring client execution.
   * **Reasoning steps** (optional).
4. **Order matters** — the outputs are append-only; earlier items may inform later items.
5. Client can store `ModelResponse.output` or individual `ModelOutput` objects for:

   * Multi-turn conversations (`previous_response_id`)
   * Agent reasoning loops
   * Token-budget management

## Usage Examples

### **Plain Text Output**

```json id="txt-out"
{
  "output": [
    {
      "type": "message",
      "role": "assistant",
      "content": [
        {
          "type": "output_text",
          "text": "Hello, world!"
        }
      ]
    }
  ]
}
```

* Represents a simple assistant reply.
* Maps directly to `ModelOutput` → `OutputMessage` → `OutputMessageContentText`.

---

### **Multimodal Output**

```json id="multi-out"
{
  "output": [
    {
      "type": "message",
      "role": "assistant",
      "content": [
        { "type": "output_text", "text": "Here is an image:" },
        { "type": "image", "image_url": "https://example.com/img.png" }
      ]
    }
  ]
}
```

* Combines text and image in a single message.
* Text appears first, image second — order preserved.

---

### **Function Call Output**

```json id="func-out"
{
  "output": [
    {
      "type": "function_call",
      "function_name": "calculate_sum",
      "arguments": { "a": 2, "b": 3 }
    }
  ]
}
```

* Signals the client to execute the function `calculate_sum`.
* Can be fed back into the next request as a `ModelInputItem`.

---

### **Reasoning Output**

```json id="reasoning-out"
{
  "output": [
    {
      "type": "reasoning",
      "content": [
        { "type": "text", "text": "Step 1: Add numbers..." },
        { "type": "text", "text": "Step 2: Return sum..." }
      ]
    }
  ]
}
```

* Provides structured reasoning steps.
* Can be used for **agent transparency** or **pruning** in long conversations.

---

## Best Practices

1. **Process outputs in order** – the API produces outputs sequentially; later outputs may depend on earlier ones.
2. **Handle all polymorphic types** – messages, reasoning, and tool calls.
3. **Preserve role information** – useful for multi-agent scenarios (`assistant`, `system`, `developer`, `tool`).
4. **Use schema-native POJOs** – do not create ad-hoc subclasses outside the schema.
5. **Store outputs for multi-turn workflows** – feed tool calls and messages back to the next `ModelRequest` as `ModelInputItem`.
6. **Annotations and logprobs** – optional metadata for each text output can be used for debugging, scoring, or custom reasoning.

---

## Key Takeaways

* **ModelResponse** is an **ordered, typed, polymorphic container** for model outputs.
* Each `ModelOutput` represents a **logical unit** of response (message, tool call, reasoning).
* Multimodal, tool, and reasoning outputs are **fully integrated**, enabling **agent-style workflows**.
* Using the **schema-native Java POJOs**, you can safely deserialize, inspect, and act upon every part of the model response.
